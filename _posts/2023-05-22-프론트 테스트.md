**테스트 목적**

개발 과정에서 생기는 변경사항들로 인해 새로운 결함이 유입되지 않았는지 확인이 가능

**테스트 종류**

1. **테스팅 트로피**

   1. **Static 테스트**: 구문오류 , 나쁜코드 스타일 등을 검증 ex)ESLint, TS(타입검사 등)
   2. **Unit 테스트**: 작은 단위를 떼어내서 분리된 환경에서 테스트 복잡한 알고리즘이 제대로 동작하는지 확인 Mocking 필요
      ex)Jest
   3. **Integration 테스트**: 어플리케이션의 여러부분들이 통합되어 제대로 상호작용 되지는지 테스트. 주로 단위 테스트보다 큰 범위(페이지)의 테스트를 의미
      ex)Jest, RTL, Enzyme
   4. **E2E 테스트**: 실제 사용자가 사용하는것과 같은 조건에서 전체 시스템을 테스트 API서버, DB등의 외부 서비스들을 모두 사용하여 통합된 시스템을 테스트 (비용 많이들고 속도가 느림)
      ex)Cypress, Selenium

   →Unit과 UI테스트의 중간 테스트인 서비스 테스트(통합테스트)

**테스팅 대상**

1. **API 서버통신**
   1. 실제 API-통합,E2E
   2. mock API-Unit,통합 (다양하게 테스트 가능)
2. **시각적 요소**: 입력값- 사용자의 액션,출력값- 사용자 액션에 따른 화면의 변화
   1. 스냅샷 테스트: HTML구조가 의도한 대로 나타나는지를 테스트 ex)Jest,StoryBook
   2. 시각적 회귀 테스트:HTML+CSS 컴포넌트가 실제로 브라우저에서 렌더링되는 모습이 의도한대로 나타나는지를 테스트 ex)StoryBook
3. 사용자 이벤트 처리 - 브라우저 환경에서 cypress

**테스팅 환경**

1. **브라우저**
   - 브라우저 호환성, 기기호환성 체크가능 →node에 비해 무거워서 초기 구동속도가 느림
     - ex) Karma, Selenium, Cypress
2. **Node.js**
   - 설치 및 실행이 간단하고 속도가 빠르다
   - 모듈 단위로 테스트가 가능
   - ex)Jest, Mocha
