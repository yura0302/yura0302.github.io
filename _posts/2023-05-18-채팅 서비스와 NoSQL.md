진행중인 프로젝트에서 채팅 서비스는 어떤 종류의 DB로 어떻게 처리되는지 궁금해서 찾아봤다.

### **NoSQL을 사용하는 이유?**

- ( 데이터 유형과 읽기/쓰기 연산의 패턴을 고려해야하는데 프로필, 친구목록, 설정 등에 관한건 관계형 DB에 넣는게 일반적.. )
  그러나 채팅데이터는 **많은 양의 메시지를 처리 + 이 많은양의 메시지들 가운데 빈번하게 사용되는건 최근 주고받은 메시지** (오래된건 안봄) .. 이런 요구사항을 만족시키기 위해서는 **키-값 저장소가 적합** - **키-값 저장소가 적합한 이유** 1. 수평적 규모 확장이 쉽다. 2. 데이터 접근 지연시간이 낮다. 3. RDBMS는 롱테일 데이터 처리를 잘 못함 + 인덱스가 커짐에 따라 접근을 처리하는 비용도 증가함 **(즉, 드물게 발생하는 데이터에 대한 접근을 처리하는데 비용이 증가한다는 뜻)**

### cf. 채팅 데이터를 RDBMS에 넣게된다면 ?

- I/O 비용 ⬆️ (I/O비용: 데이터를 넣고 빼는데 리소스 (CPU,DISK,Memory)를 사용하는 용량이 크다는 뜻)
- Nosql은 상대적으로 IO비용이 적다. (관계정의 필요없이 키,밸류로만 저장하면 되니까)

### cf2. 어떤 NoSQL을 사용하는것이 적합할까 ?

| Redis                                                                                | MongoDB                                                                                                                                                                                | Cassandra                                                                                                                     |
| ------------------------------------------------------------------------------------ | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------- |
| 예측 가능한 데이터베이스 크기로 빠르게 변화하는 데이터용(대부분 메모리에 맞아야 함). | 동적 쿼리가 필요한 경우. 매핑/축소 기능이 아닌 인덱스 정의를 선호하는 경우. 큰 DB에서 좋은 성능이 필요한 경우. CouchDB를 원했지만 데이터가 너무 많이 변경되어 디스크를 가득 채운 경우. | 너무 커서 서버에 맞지 않는 데이터를 저장해야 하지만 친숙하고 친숙한 인터페이스가 필요한 경우. (대용량 데이터 저장+ 친숙한 UI) |
| “실시간” 서비스에 주로 사용,주가분석예측 등                                          | 대부분의 경우 MySQL 또는PostgreSQL로 수행하지만 미리 정의된 열을 사용하면 실제로 방해가 됩니다.                                                                                        | 시간별, 브라우저별, IP별 등의 조회수를 계산하는 웹 분석 트랜잭션 로깅. 거대한 센서 어레이에서 데이터 수집.                    |
